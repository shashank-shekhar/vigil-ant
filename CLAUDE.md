## Project Overview

Vigil-ant is a macOS menu bar app that monitors GitHub repositories for CI/CD build failures. It uses SwiftUI with `MenuBarExtra` (`.window` style), targets macOS 14+, and requires Swift 5.9+.

## Build & Test Commands

```bash
# Test packages (the main development loop)
cd Packages/GitHubKit && swift test
cd Packages/CIStatusKit && swift test

# Run a single test
cd Packages/GitHubKit && swift test --filter GitHubAPIClientTests
cd Packages/CIStatusKit && swift test --filter StatusPollerTests

# Clean build (fixes stale cache issues after model changes)
cd Packages/CIStatusKit && swift package clean && swift test

# Build the app (Xcode)
xcodebuild build -project Vigilant/Vigilant.xcodeproj -scheme Vigilant

# Regenerate menu bar status icons
swift scripts/generate-menu-bar-icons.swift

# Local mock server (no GitHub API calls needed)
python3 scripts/mock-github-server.py "3s2f1r1n" "3s2f1r" "10f"
# Then set GITHUB_BASE_URL=http://localhost:8787 in Xcode scheme env vars

# Build release DMG (optional --notarize flag)
scripts/build-release.sh
```

The app itself is built and run through Xcode. The two local Swift packages can be tested independently via CLI. CI runs on macOS-15 via `.github/workflows/ci.yml` (tests both packages, then builds the app). Releases are automated via `.github/workflows/release.yml` triggered by `v*` tags.

## Architecture

**Two local Swift packages + one app target:**

- **GitHubKit** (`Packages/GitHubKit/`) — GitHub API client, device flow auth, data models. No external dependencies. Key types: `GitHubAPIClient` (actor), `DeviceFlowManager` (device code auth), `Account`, `Repository`, `WorkflowRun`, `CombinedStatus`.

- **CIStatusKit** (`Packages/CIStatusKit/`) — Polling engine and status aggregation. Depends on GitHubKit. Key types: `StatusPoller` (actor, concurrent polling), `StatusAggregator` (@Observable, merges Actions + commit status), `BuildStatus` (severity-based merge logic).

- **App** (`App/`) — SwiftUI views, app state, utilities. `AppState` (@Observable, @MainActor) orchestrates polling, persistence (UserDefaults + Keychain), and notifications. Not a Swift package — built via Xcode project at `Vigilant/Vigilant.xcodeproj`.

**Data flow:** `AppState` owns `StatusAggregator` and `StatusPoller` → poller fetches via `GitHubAPIClient` → results flow into aggregator → SwiftUI views observe aggregator.

## Key Patterns

- **@Observable** (not ObservableObject/Combine) for reactive state. No `@State` wrapper needed at App level for @Observable types.
- **Actors** for thread-safe API client and poller. `@MainActor` for UI state.
- **ETag caching** in `GitHubAPIClient.getWithETag()` — returns nil on 304 Not Modified to avoid wasting rate limit. Exponential backoff (max 2 retries) for transient 5xx/network errors.
- **Dependency injection** in `StatusPoller` via `RepoFetcher` closure for testability. `TokenRefresher` callback lets AppState provide refresh logic.
- **`hasWorkflows` flag** on `Repository` — repos without GitHub Actions are filtered out of polling and UI. Re-checked every 5 poll cycles. Failure on re-check preserves existing flag. Migration logic in `AppState.init` handles backward compat.
- **Dependabot filtering** — workflow runs with paths containing "dependabot" are excluded.
- **Ghost pending fix** — GitHub returns `"pending"` with 0 statuses for repos with no CI; treated as no data.
- **Auth** uses a GitHub App with Device Flow (`DeviceFlowManager`) — fine-grained permissions (Actions read, Commit statuses read, Metadata read). No client secret or callback server needed.
- **Tokens** stored in Keychain (`KeychainHelper`), never in UserDefaults. On refresh, the refresh token is saved _before_ the access token — if the access token save fails, the old refresh token remains valid.
- **Polling phases** in `StatusPoller`: (1) parallel fetch all repos, (2) detect 401s and attempt token refresh, (3) retry repos whose tokens were refreshed, (4) update aggregator on MainActor.
- **Status severity** — `BuildStatus` uses numeric severity (unknown=0, success=1, building=2, pending=3, failure=4). Merge takes the worst of Actions + Commit Status. Aggregation at repo, account, and global levels follows the same "worst wins" rule.
- **Menu bar icons** are pre-rendered template images in the Asset Catalog (status-ok, status-1 through status-9, status-10plus), generated by `scripts/generate-menu-bar-icons.swift`.

## Testing

Tests use Swift's native `Testing` framework (`@Test` macros, not XCTest). `StatusPoller` tests require `hasWorkflows: true` on test repos or they get filtered out.

## Environment

GitHub App Client ID and Sparkle public key are injected at build time via `Secrets.xcconfig` → Info.plist `$(GH_CLIENT_ID)` / `$(SPARKLE_PUBLIC_KEY)` substitutions. `OAuthConfig.swift` reads the plist value with a fallback to the `VIGIL_ANT_CLIENT_ID` environment variable (useful for CI). No client secret needed (device flow). The app runs as a menu bar agent (`LSUIElement = YES` in Info.plist).

## Persistence

- **UserDefaults** — accounts, repositories, cached statuses, poll interval, onboarding state, data schema version.
- **Keychain** — access tokens and refresh tokens (never UserDefaults).
- **Schema versioning** — `dataSchemaVersion` in UserDefaults with migration logic in `AppState.init`.
- **Network monitor** — `NetworkMonitor` (@Observable) watches NWPathMonitor; triggers automatic refresh when connectivity is restored.
